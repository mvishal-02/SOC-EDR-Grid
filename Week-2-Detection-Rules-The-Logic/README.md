# Week 2 – Detection Rules Implementation

# Overview

Week 2 focuses on implementing **detection logic** within the SOC monitoring infrastructure that was deployed in Week 1.

During this phase, the Wazuh detection engine was configured to identify suspicious system behavior and potential security threats. This was achieved through **File Integrity Monitoring (FIM), custom log decoders, detection rules, and vulnerability scanning**.

These enhancements strengthened the monitoring capability of the Sentient Shield EDR Grid by enabling the system to detect unauthorized file changes, suspicious log activity, and known software vulnerabilities.

# Objective

The objective of Week 2 was to configure advanced detection mechanisms within the Wazuh-based SOC infrastructure.

# Key tasks included:

Configuring File Integrity Monitoring (FIM) to monitor sensitive directories

Creating custom XML decoders for proprietary application logs

Implementing custom detection rules to generate high-severity alerts

Enabling the Vulnerability Detector module

Verifying that alerts are generated when monitored files are modified

These configurations enabled the system to detect unauthorized file modifications and suspicious activity in real time.

File Integrity Monitoring Configuration

File Integrity Monitoring (FIM) was configured on the Kali Linux agent and on WIndows 10 agent to monitor sensitive directories and security-critical system files.

This feature allows Wazuh to detect unauthorized modifications, deletions, or additions to important system files.

# Configuration file location:

/var/ossec/etc/ossec.conf

# Example configuration:

<syscheck>
  <directories check_all="yes">/etc</directories>
  <directories check_all="yes">/var/www</directories>
</syscheck>

# Explanation:

/etc → Contains critical Linux system configuration files

/var/www → Web server directory that could contain malicious uploads

Whenever a change occurs in these directories, Wazuh generates an alert.

# Custom XML Decoder Implementation

To analyze proprietary application logs, a custom XML decoder was created on the Wazuh Manager.

Configuration file location:

/var/ossec/etc/decoders/local_decoder.xml

# Command used:

sudo nano /var/ossec/etc/decoders/local_decoder.xml

# Decoder configuration:

<decoder name="custom_app_log">
 <prematch>^\[ALERT\]</prematch>
 <regex>USER=(\w+)\sACTION=(\w+)\sFILE=(\S+)</regex>
 <order>user action file</order>
</decoder>

This decoder extracts the following information from application logs:

Username performing the action

Action executed by the user

File affected by the operation

This enables Wazuh to properly interpret logs generated by proprietary applications.

# Custom Detection Rule Implementation

After creating the decoder, a custom rule was implemented to generate high-severity alerts when suspicious activity is detected.

# Configuration file:

/var/ossec/etc/rules/local_rules.xml

# Command used:

sudo nano /var/ossec/etc/rules/local_rules.xml

# Rule configuration:

<rule id="100200" level="12">
 <if_matched_sid>custom_app_log</if_matched_sid>
 <description>Critical action in proprietary application log</description>
 <mitre>T1046</mitre>
</rule>

# Explanation:

Rule ID → Unique identifier for the rule

Level 12 → High severity alert

MITRE Mapping → T1046 (Network Service Discovery)

After adding the rule, the Wazuh Manager service was restarted:

sudo systemctl restart wazuh-manager

# Enabling the Vulnerability Detector
The Vulnerability Detector module was enabled to identify known vulnerabilities in installed software packages.

# Configuration file:

/var/ossec/etc/ossec.conf

# Command used:

sudo nano /var/ossec/etc/ossec.conf

# Configuration added:

<vulnerability-detector>
 <enabled>yes</enabled>
 <interval>5m</interval>
 <run_on_start>yes</run_on_start>
</vulnerability-detector>

# Explanation:

enabled → Activates vulnerability scanning

interval → Scans for vulnerabilities every 5 minutes

run_on_start → Runs a scan when the manager starts

# Apply the configuration:

sudo systemctl restart wazuh-manager

# Detection Testing

To verify that the detection mechanisms were working correctly, several tests were performed.

# Test 1 – Modifying a Critical System File

A change was made to the /etc/passwd file to simulate unauthorized system modification.

# Command used:

sudo nano /etc/passwd

Since this file is monitored by File Integrity Monitoring, the modification triggered an alert in the Wazuh dashboard.

# Test 2 – Simulating a Web Server Attack

A suspicious file was created in the monitored web directory.

# Command used:

sudo touch /var/www/malicious.php

This simulated a possible web shell upload attack.

The creation of this file triggered an alert indicating that a new file had been added to a monitored directory.
